# SCHEMA.md — The Content Treaty

This is the **single most important contract in the project.** It is the only place the AI-coded engine (Pipeline A) and the AI-authored world (Pipeline B) meet (`GOAL.md §6`). If content validates against these schemas, the engine runs it — no matter who wrote it.

These are written as **Zod** schemas (runtime validation + inferred TS types). The same definitions export to **JSON Schema** (via `zod-to-json-schema`) to constrain LLM structured output in the offline pipeline. **Define once, use on both sides.**

Location: `packages/content-schema/src/`.

---

## 0. Design rules for the schema itself

- **Branded IDs everywhere.** A `QuestId` is not interchangeable with an `NpcId`. This catches cross-wiring at compile time.
- **Discriminated unions for variants.** Objectives, effects, and conditions all use a `kind` discriminator so `applyEvent` can switch exhaustively.
- **No free-form behavior.** Content can only express things the engine already understands. Content cannot smuggle in code. If the world needs a new verb, the *agent* adds it to the engine + schema (a ticket), then content can use it.
- **References are IDs, validated at load.** Content says `giverNpcId: "npc.varga"`; the loader proves `npc.varga` exists.
- **Conditions are a tiny safe expression language**, not eval. See §7.

## 1. IDs and primitives (`ids.ts`)

```ts
import { z } from "zod";

// The prefix is enforced at RUNTIME, not just at compile time. Content loads from
// JSON, so without this both an NpcId and a QuestId could parse the same string.
const brand = <B extends string>(prefix: B) =>
  z.string()
   .regex(new RegExp(`^${prefix}\\.[a-z0-9_]+(\\.[a-z0-9_]+)*$`), `expected id like "${prefix}.example"`)  // underscores allowed in every segment (matches the canonical ids in §9/§11)
   .brand<B>();

export const NpcId      = brand("npc");
export const QuestId    = brand("quest");
export const LocationId = brand("location");
export const FactionId  = brand("faction");
export const ItemId     = brand("item");
export const DialogueId = brand("dialogue");
export const FlagId     = brand("flag");          // a world-state boolean/number key

export type NpcId = z.infer<typeof NpcId>;
// …etc. IDs look like "npc.varga", "quest.the_warehouse", "faction.ashfall_syndicate".
```

## 2. Locations (`location.ts`)

A location is a place the player can be: a district, a building interior, a room. Top-down, vector-described.

> **Who authors vector art:** LLMs are poor at plotting recognizable shapes via raw coordinates. So **`VectorShape` arrays are hand-authored or produced by a layout helper/tool — never free-generated by the content pipeline.** The pipeline (Pipeline B) authors *text* (bios, dialogue, quests, mood) and *references*; it may place pre-defined shapes by parameter, but it does not invent vertex tangles. When richer art is wanted, that is the future sprite/AI-image layer behind the `Renderer` interface, not LLM-generated paths here. (`prefer rect/circle; use path sparingly and only when hand-tuned`.)

```ts
export const Vec2 = z.object({ x: z.number(), y: z.number() });

export const VectorShape = z.discriminatedUnion("kind", [
  z.object({ kind: z.literal("rect"), at: Vec2, w: z.number(), h: z.number(), fill: z.string(), stroke: z.string().optional() }),
  z.object({ kind: z.literal("circle"), at: Vec2, r: z.number(), fill: z.string(), stroke: z.string().optional() }),
  z.object({ kind: z.literal("path"), points: z.array(Vec2).min(2), stroke: z.string(), fill: z.string().optional() }),
]);

export const Exit = z.object({
  at: Vec2,                       // where the door/edge is
  toLocationId: LocationId,       // where it leads
  spawnAt: Vec2,                  // where the player appears on arrival
  label: z.string().max(40),      // "Bar interior", "Back alley"
  requires: z.array(Condition).default([]), // gated exits (locked doors, etc.)
});

export const Location = z.object({
  id: LocationId,
  name: z.string().max(60),
  mood: z.string().max(280),      // short prose shown on entry; sets tone
  bounds: z.object({ w: z.number(), h: z.number() }),
  art: z.array(VectorShape).max(120),      // the vector scene
  exits: z.array(Exit).default([]),
  npcSpawns: z.array(z.object({ npcId: NpcId, at: Vec2 })).default([]),
  ambientText: z.array(z.string().max(160)).default([]), // optional flavor lines
});
export type Location = z.infer<typeof Location>;
```

## 3. NPCs (`npc.ts`)

The unit the content pipeline produces most. Deep characterization, but all behavior is data the engine understands.

```ts
export const Npc = z.object({
  id: NpcId,
  name: z.string().max(60),
  faction: FactionId.optional(),
  appearance: z.object({          // vector-described, upgradeable to sprites later
    bodyColor: z.string(),
    accentColor: z.string(),
    silhouette: z.enum(["humanoid", "tall", "stocky", "cloaked", "mech", "beast"]),
  }),
  // The soul of the character — used by the pipeline for consistency, shown via dialogue:
  bio: z.object({
    role: z.string().max(80),          // "dock-side fixer", "disgraced medic"
    backstory: z.string().max(1200),   // canon; the world bible references this
    wants: z.string().max(240),        // current goal — drives quests
    fears: z.string().max(240),
    voice: z.string().max(240),        // how they speak; guides dialogue authoring
    secrets: z.array(z.string().max(240)).max(5).default([]),
  }),
  dialogueId: DialogueId,              // entry-point Ink story for talking to them
  // Optional combat stats — present only for NPCs that can be the target of a `defeat`
  // objective. Additive/optional so existing packs and saves stay valid (SCHEMA.md §10).
  combat: z.object({ hp: z.number().int().positive() }).optional(),
  // Where this NPC lives; the app spawns it there on entry. Decouples placement from the
  // location definition, so a pack can place its NPCs without the location listing them.
  homeLocationId: LocationId.optional(),
  // Reactivity: lines/behaviors that change based on world state.
  reactsTo: z.array(z.object({
    when: z.array(Condition).min(1),
    setsFlags: z.array(FlagEffect).default([]),
    overrideDialogueId: DialogueId.optional(),  // a different conversation once X is true
  })).default([]),
});
export type Npc = z.infer<typeof Npc>;
```

## 4. Quests (`quest.ts`)

A quest is **offered** when conditions hold, **completed** when objectives are met, and produces **effects** that change the world. Multiple solutions are expressed by allowing different objective sets / branches — agency is a data property, not a script.

```ts
export const Objective = z.discriminatedUnion("kind", [
  z.object({ kind: z.literal("talk_to"),  npcId: NpcId }),
  z.object({ kind: z.literal("reach"),     locationId: LocationId }),
  z.object({ kind: z.literal("retrieve"),  itemId: ItemId, count: z.number().int().positive() }),
  z.object({
    kind: z.literal("skill_check"),
    skill: z.enum(["persuade","sneak","force","tech"]),
    dc: z.number().int().min(1).max(20),
    retryable: z.boolean().default(false),     // default: a blown check commits the consequence
    onFail: z.array(Effect).default([]),       // FIRST-CLASS failure: foreclose branch, raise
                                               // suspicion, fall through to combat, change a line…
  }),
  z.object({ kind: z.literal("defeat"),    npcId: NpcId }),
  z.object({ kind: z.literal("set_flag"),  flag: FlagId, to: z.union([z.boolean(), z.number(), z.string()]) }),
]);

// A "branch" is one viable way to satisfy the quest. This is how we get
// "talk OR sneak OR fight" without scripting — each branch is a path through state.
export const Branch = z.object({
  id: z.string().max(40),
  label: z.string().max(80),                  // "Talk your way in", "Slip past the guards"
  objectives: z.array(Objective).min(1).max(6),
  onComplete: z.array(Effect).default([]),
  onFail: z.array(Effect).default([]),        // applied if this branch becomes impossible
});
```

> **Why failure is first-class:** in the genres we're invoking (Disco Elysium, BG3), *interesting failure is where agency lives.* A failed persuade should be able to fall through to combat, raise suspicion, or change a later line — not just leave an objective unmet. Runtime semantics for failure (when a branch becomes `failed`, retry rules, idempotency) are defined in `WORLD_STATE.md §2–§3`. Skill checks resolve against the **player's actual skills + situational modifiers** earned in play (`WORLD_STATE.md §3`), so "the talk solution draws on what you learned in the bar" is mechanically real, not flavor text.


export const Quest = z.object({
  id: QuestId,
  title: z.string().max(80),
  giverNpcId: NpcId.optional(),
  summary: z.string().max(400),               // journal entry
  offerWhen: z.array(Condition).default([]),  // gates availability
  branches: z.array(Branch).min(1).max(5),    // ≥1; multiple = player agency
  onAnyComplete: z.array(Effect).default([]), // effects regardless of branch
  rewards: z.object({
    credits: z.number().int().nonnegative().default(0),
    items: z.array(z.object({ itemId: ItemId, count: z.number().int().positive() })).default([]),
    reputation: z.array(z.object({ factionId: FactionId, delta: z.number().int() })).default([]),
  }),
});
export type Quest = z.infer<typeof Quest>;
```

## 5. Effects (`effect.ts`) — the only ways content can change the world

Effects map 1:1 to engine `Event`s. Content cannot do anything the engine doesn't already implement.

```ts
export const FlagEffect = z.object({
  kind: z.literal("set_flag"), flag: FlagId, to: z.union([z.boolean(), z.number(), z.string()]),
});

export const Effect = z.discriminatedUnion("kind", [
  FlagEffect,
  z.object({ kind: z.literal("adjust_reputation"), factionId: FactionId, delta: z.number().int() }),
  // give_item count may be signed (negative = remove), but applyEvent REJECTS any apply
  // that would drive inventory below zero. See WORLD_STATE.md §5 (conservation & bounds).
  z.object({ kind: z.literal("give_item"),  itemId: ItemId, count: z.number().int().refine(n => n !== 0, "count must be non-zero") }),
  // grants a temporary situational skill modifier — e.g. a bar tip granting +2 persuade
  // vs the guard. This is how "what you learned" feeds the skill-check math (WORLD_STATE.md §3).
  z.object({ kind: z.literal("modify_skill"), skill: z.enum(["persuade","sneak","force","tech"]), delta: z.number().int() }),
  z.object({ kind: z.literal("start_quest"), questId: QuestId }),
  z.object({ kind: z.literal("unlock_exit"), locationId: LocationId, exitIndex: z.number().int().nonnegative() }),
  z.object({ kind: z.literal("set_npc_dialogue"), npcId: NpcId, dialogueId: DialogueId }),
  z.object({ kind: z.literal("show_text"), text: z.string().max(400) }),
  // bribe a faction: spend `cost` credits (only if affordable, enforced in applyEvent) to shift
  // standing by `standing`. Added in T-16 — the worked example of the "extending the vocabulary" note below.
  z.object({ kind: z.literal("bribe_faction"), factionId: FactionId, cost: z.number().int().positive(), standing: z.number().int() }),
]);
export type Effect = z.infer<typeof Effect>;
```

> **Bounds belong in `applyEvent`, not the schema:** the schema permits signed deltas; the engine's single `applyEvent` chokepoint clamps reputation to `[-100, 100]` and forbids negative inventory (`WORLD_STATE.md §5`). This is what makes the "credits are conserved / no negative inventory" invariant tests enforceable rather than aspirational.

> **Extending the world's vocabulary:** when curated content wants a verb that doesn't exist (e.g. "bribe a faction"), that is a signal to the **agent** to add a new `Effect`/`Event` pair + system to the engine (a ticket), *then* content can use it. The schema grows deliberately, never by smuggling logic into data.

## 7. Conditions (`condition.ts`) — a tiny, safe expression language

Conditions gate quests, exits, and NPC reactions. **No `eval`.** A condition is a small typed tree the engine evaluates against `World`.

```ts
export const Condition: z.ZodType<Condition> = z.lazy(() =>
  z.discriminatedUnion("kind", [
    z.object({ kind: z.literal("flag_is"), flag: FlagId, equals: z.union([z.boolean(), z.number(), z.string()]) }),
    z.object({ kind: z.literal("reputation_at_least"), factionId: FactionId, value: z.number().int() }),
    z.object({ kind: z.literal("has_item"), itemId: ItemId, count: z.number().int().positive().default(1) }),
    z.object({ kind: z.literal("quest_completed"), questId: QuestId }),
    z.object({ kind: z.literal("credits_at_least"), amount: z.number().int().nonnegative() }),
    z.object({ kind: z.literal("skill_at_least"), skill: SkillName, value: z.number().int() }),
    z.object({ kind: z.literal("not"), of: Condition }),
    z.object({ kind: z.literal("all"), of: z.array(Condition).min(1) }),
    z.object({ kind: z.literal("any"), of: z.array(Condition).min(1) }),
  ])
);
export type Condition =
  | { kind: "flag_is"; flag: FlagId; equals: boolean | number | string }
  | { kind: "reputation_at_least"; factionId: FactionId; value: number }
  | { kind: "has_item"; itemId: ItemId; count: number }
  | { kind: "quest_completed"; questId: QuestId }
  | { kind: "credits_at_least"; amount: number }
  | { kind: "skill_at_least"; skill: SkillName; value: number } // a passive check (Disco-Elysium style): gates content on World.player.skills
  | { kind: "not"; of: Condition }
  | { kind: "all"; of: Condition[] }
  | { kind: "any"; of: Condition[] };
```

The engine's `conditions/conditions.ts` implements `evaluate(world, condition): boolean` — pure, total, exhaustive over the union. Adding a new condition kind is: one schema entry + one switch arm + one test.

## 8. The content pack envelope (`pack.ts`)

Content ships in **packs**: a manifest + arrays of entities. Generated packs carry provenance so curation is auditable.

```ts
export const Provenance = z.object({
  authoredBy: z.enum(["human", "pipeline"]),
  models: z.array(z.string()).default([]),       // e.g. ["claude-opus-4.x","gpt-5.x"]
  promptHash: z.string().optional(),
  curatedBy: z.string().optional(),              // human approver
  approvedAt: z.string().datetime().optional(),
});

// Dialogue is FIRST-CLASS content. Without this, referential integrity cannot
// validate the DialogueId references that NPCs and effects point at.
export const DialogueAsset = z.object({
  id: DialogueId,
  format: z.literal("ink-json"),
  inkVersion: z.string(),
  sourceHash: z.string(),          // hash of the .ink source it compiled from
  compiled: z.unknown(),           // inkjs compiled JSON; shape owned by narrative-ink adapter
  declaredVars: z.array(z.string()).default([]),  // story vars this dialogue mirrors to flags
});

export const ContentPack = z.object({
  id: z.string().regex(/^pack\.[a-z0-9_]+$/),
  version: z.string(),                            // semver
  title: z.string().max(80),
  dependsOn: z.array(z.string()).default([]),     // other pack ids
  provenance: Provenance,
  locations: z.array(Location).default([]),
  npcs: z.array(Npc).default([]),
  quests: z.array(Quest).default([]),
  factions: z.array(Faction).default([]),
  items: z.array(ItemTemplate).default([]),
  dialogues: z.array(DialogueAsset).default([]),  // compiled Ink, referenced by DialogueId
  storylets: z.array(Storylet).default([]),       // quality-based reactive content units (SPEC-11; see below)
  assertions: z.array(CanonAssertion).default([]),// structured canon facts (CONTENT_PIPELINE §6)
});
export type ContentPack = z.infer<typeof ContentPack>;
```

`content-loader` loads packs in dependency order, validates each with Zod, then runs a **referential-integrity pass**: every `NpcId`/`QuestId`/`ItemId`/`LocationId`/`FactionId`/`DialogueId` referenced anywhere must resolve to a defined entity. Any dangling reference fails the load with a precise error. This is how an AI-authored pack can never silently break the game.

**Layering corollary — a base pack must load in isolation.** The loader runs against whatever *subset* of packs it is given (the live app loads one set; tests such as the replay-fuzz load a single base pack **alone**), so referential integrity is checked against that subset, not the whole catalog. A base pack must therefore be **self-contained**: if it carries an `exit.toLocationId` (or any other reference) pointing at a location, that **location must live in the base pack itself** (shared world *geography*) — with only the *content* that fills it (NPCs, quests, dialogues, storylets) overlaid from a dependent pack via `dependsOn`. Put the geography in the overlay instead and the base pack fails to load on its own. (Worked example: the Drip Market's two locations live in `pack.opening`; `pack.drip_market` overlays only the vendors, quest, and barks — SPEC-35.)

**Canon assertions** (`assertion.ts`, additive + ENGINE-IGNORED) are structured facts — `subject → predicate → (object | status state)`, with an optional `since` epoch for status-over-time. The simulation never reads them; the offline **canon assertion graph** (`CONTENT_PIPELINE.md §6`) queries them to catch *semantic* contradictions the ID-level check can't (a broke patron who funds a faction, a dead NPC still placed in the world, allies who are also enemies). Predicates: `member_of`, `allied_with`, `enemy_of`, `funds`, `located_in`, `status`, `fact`.

**Storylets** (`storylet.ts`, SPEC-11 — additive, salience-selected) are quality-based content units: `preconditions` (a `Condition[]`, §7), an integer `salience`, `tags`, `content` (`{ dialogueId? | ambient? }`), and `effects` (§6). The engine's `storyletSystem` filters storylets whose preconditions hold, ranks them by `salience`, and emits the highest-salience candidates as a `TriggerStorylet` event (ties broken by the seeded RNG **inside** `applyEvent`, so selection stays replay-deterministic). Design rule: salience drives **reactive/ambient** content (barks, flavor that responds to state); **main-plot beats stay gated behind explicit quest flags, never salience** — this avoids a player incidentally satisfying a dramatic beat's preconditions out of order.

      ],
      "onComplete": []
    },
    {
      "id": "force", "label": "Just take it",
      "objectives": [
        { "kind": "reach", "locationId": "location.warehouse_door" },
        { "kind": "defeat", "npcId": "npc.warehouse_guard" },
        { "kind": "reach", "locationId": "location.warehouse_floor" }
      ],
      "onComplete": [
        { "kind": "adjust_reputation", "factionId": "faction.ashfall_syndicate", "delta": -15 }
      ]
    }
  ],
  "onAnyComplete": [
    { "kind": "give_item", "itemId": "item.encrypted_drive", "count": 1 },
    { "kind": "set_flag", "flag": "flag.has_drive", "to": true }

```

Three viable solutions, each just a path through world state, each with different consequences — **that is player agency expressed as data** (`GOAL.md §4`). A failed persuade isn't a dead end: it sets `flag.guard_suspicious` (which later dialogue and the `force` branch can read) — *interesting failure* as data. No engine change is needed to ship this; the engine already understands `reach`, `skill_check`, `defeat`, and these effects.

## 10. Evolving the schema (the long-term survival rule)

The world grows for years; the schema will change. The process is bounded so old packs and saves don't silently break:

1. **New verbs follow the "add a system" path** (`AGENT_GUIDES.md` Recipe 1): a new objective kind / effect / condition is added to the union here **and** handled in the engine, together, in one ticket.
2. **Deprecate, don't delete.** Mark a field deprecated in Zod (`.describe("DEPRECATED: …")`), keep supporting it for one major version, then drop it.
3. **Version + migrate.** Any change to `ContentPack`, `World`, or `ReplayLog` format increments that format's version (`WORLD_STATE.md §7`) and ships a migration step + a migration test from the prior version.
4. **Every schema-change ticket's acceptance criteria must include "migration path for existing packs/saves."** A migration script template lives at `tools/migrate/`.
